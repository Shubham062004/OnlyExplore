# Architecture & Design Decisions

This document outlines the high-level architecture, core design decisions, and extensibility patterns of the **OnlyExplore** application.

## 1. High-Level Architecture

OnlyExplore is built as a full-stack, serverless-ready web application using **Next.js 15 (App Router)**. It follows a monolithic repository structure where both the frontend client and the backend API endpoints reside in the same codebase. 

The architecture leverages:
- **Server Components (RSC)** for initial data fetching and rendering to improve performance and SEO.
- **Client Components** for highly interactive UI elements (e.g., interactive maps, AI chat interfaces).
- **Serverless API Routes** (`src/app/api/...`) for handling webhooks, form submissions, and authentication.

## 2. Core Technologies & Rationale

- **Next.js 15 & React 18**: Chosen for seamless Server-Side Rendering (SSR), efficient routing, and built-in API handling. The App Router paradigm ensures clean separation between layouts and pages.
- **Google Genkit**: Used for AI workflow orchestration. Genkit allows us to structure, trace, and manage complex LLM interactions (like generating destination guides and conversational planning) more effectively than raw API calls.
- **MongoDB (Mongoose)**: Selected as the primary database for its flexible document model, which pairs perfectly with the dynamic, unpredictable JSON structures generated by AI.
- **NextAuth.js v5**: Provides a secure, unified abstraction for handling multiple authentication methods (OAuth, Magic Links, Credentials) using secure, HTTP-only JWTs.
- **Tailwind CSS & shadcn/ui**: Enables rapid, utility-first styling with accessible, unstyled component primitives that can be heavily customized.

## 3. Directory Structure & Separation of Concerns

The codebase is organized to enforce a strict separation of concerns:

- `src/app/`: The routing layer. Contains pages, layouts, and API routes. Data fetching happens at the highest possible level in server components.
- `src/components/`: Pure presentation layer. Components are decoupled from business logic and data fetching wherever possible.
- `src/lib/`: Core utilities. Contains database connection logic (`db.ts`), strict environment validation (`env.ts`), and third-party initializations (`stripe.ts`).
- `src/models/`: The data layer. Contains Mongoose schemas that strictly define the shape of our MongoDB collections.
- `src/ai/`: The AI orchestration layer. Isolates Genkit flows, prompts, and Zod schemas used to validate AI outputs.

## 4. Key Design Decisions

### 4.1. Intelligent Caching via MongoDB TTL
Instead of querying third-party APIs (like Unsplash for images) on every request, we implemented a caching layer directly in MongoDB. 
**Decision**: Use MongoDB's Time-To-Live (TTL) indexes. 
**Benefit**: Automatically expires old cache documents, ensuring data freshness while significantly reducing external API rate-limiting risks and latency.

### 4.2. Structured AI Responses
AI-generated content is inherently unpredictable.
**Decision**: Force the LLM to output structured JSON and validate it using **Zod** within the Genkit flows.
**Benefit**: The application UI never breaks due to malformed AI text. If the AI hallucinates a structure, Zod catches it, and the flow can retry or fallback gracefully.

### 4.3. Premium Feature Gating
**Decision**: Handle Stripe subscription verification at the server level before rendering pages or returning API data.
**Benefit**: Ensures that premium features (like advanced itinerary generation or interactive maps) cannot be bypassed by manipulating client-side state.

### 4.4. Decoupled Authentication
**Decision**: Removed CAPTCHA and streamlined the auth flow, delegating session management entirely to NextAuth.
**Benefit**: Reduces friction for user onboarding while maintaining security via NextAuth's CSRF protection and encrypted JWTs.

## 5. Extensibility Patterns

The architecture is designed to grow. Here is how developers can extend the system:

### Adding New AI Workflows
To add a new AI feature (e.g., "Flight Price Predictor"):
1. Define the input/output Zod schemas in `src/ai/schemas.ts`.
2. Create a new flow in `src/ai/flows.ts` using Genkit.
3. Call the flow from a Server Action or API Route in `src/app/`.

### Expanding the Data Model
Because we use MongoDB, extending schemas is non-destructive.
1. Add new fields to the relevant Mongoose schema in `src/models/`.
2. Typescript interfaces should be updated to reflect the schema changes.
3. No complex database migrations are required for additive changes.

### Introducing New Third-Party APIs
New integrations (e.g., a new restaurant API) should be wrapped in utility functions inside `src/lib/`. 
- **Rule**: Never call third-party APIs directly from a React component. Always route them through a server component or an API route to protect API keys.

### Modifying Subscription Tiers
Stripe webhooks are fully integrated into `src/app/api/webhook/stripe/route.ts`. To add a new tier:
1. Create the product/price in the Stripe Dashboard.
2. Add the new Price ID to the `.env` file and validate it in `src/lib/env.ts`.
3. Update the conditional logic in the application that checks `user.subscriptionTier`.
